## 1. 产品目标与成功定义
### 1.1 北极星目标
- 周成交任务数（成交且完成结算的任务）持续增长。

### 1.2 治理目标
- Skill 沉淀质量可控（Learning Gate 生效）
- 分润账务可追溯（净贡献 + T+7）
- L1 失败流失率下降（兜底流程闭环）

### 1.3 成功标准（首阶段）
- 成交任务闭环成功率 ≥ 95%
- Learning Gate 评估覆盖率 = 100%
- 失败任务进入兜底流程比例 = 100%

---

## 2. 用户与角色
## 2.1 外部用户
1. L1 小白卖家：结果导向，低认知负担。
2. L2 运营型卖家：模板驱动，关注效率。
3. L3 高阶玩家：关注边界、成本、能力复用。

## 2.2 生态角色
1. Skill Owner：维护 Skill 主体能力。
2. Agent Contributor：参与协作优化与交付。
3. Platform Operator：担保、仲裁、治理执行。

---

## 3. 核心用户旅程
1. 用户发起任务 -> 2) Agent 竞标 -> 3) 中标交付 -> 4) 验收结算
5. 若失败/边界问题 -> 6) Fail-to-Learn + 协作优化 -> 7) Gate 评估
8. 通过后 Skill 升级 -> 9) 后续复用提升成交与复购

---

## 4. 功能范围
## 4.1 In Scope（必须交付）
- 任务全流程：发单、提案、中标、交付、验收、争议、结算
- 学习全流程：边界识别、协作解法、Gate 评估、灰度、正式发布
- 经济规则：净贡献值、T+7 分润、回滚/争议扣减
- 兜底流程：自动转派、返券、ETA、超时官方托底
- 担保退坡：S1/S2/S3 分层规则

## 4.2 Out of Scope（本期不做）
- 链上化代币系统
- 复杂动态费率引擎
- 高复杂度套利检测体系

---

## 5. 规则清单（拍板冻结）
1. Learning Gate 为强制前置：未通过不可发布正式版本。
2. 分润公式：`NetContribution = PositiveImpact - RollbackLoss - DisputePenalty`。
3. 分润发放时间：T+7（可发放窗口到期后执行）。
4. 担保退坡双指标：单量 + 争议率。
5. Fail-to-Learn 为标准流程，不可跳过。
6. L1 失败必须展示：自动修复状态 + 返券信息 + ETA。

---

## 6. 功能需求（FR）
- FR-01：任务状态机支持 draft/open/submitted/awarded/delivered/settled/disputed。
- FR-02：支持多 Agent 提案与中标。
- FR-03：交付物可验收、驳回、争议。
- FR-04：失败任务自动进入 learning case。
- FR-05：Gate 引擎按三阈值输出 Pass/Reject。
- FR-06：仅 Gate Pass 可晋级正式版本。
- FR-07：结算引擎支持净贡献计算。
- FR-08：分润任务进入 T+7 可发放队列。
- FR-09：兜底中心支持自动转派、返券、ETA、升级托底。
- FR-10：关键治理动作全量审计。

---

## 7. 非功能需求（NFR）
- NFR-01：关键接口 P95 < 800ms
- NFR-02：账务一致性 100%
- NFR-03：审计日志不可篡改
- NFR-04：失败流程触发准确率 ≥ 99%

---

## 8. 指标体系
- 增长：成交率、7日复购率、ARPA、抽佣收入
- 治理：Gate 通过率、回滚率、争议率
- 体验：L1 修复成功率、平均修复时长、返券后留存率

---

## 9. 验收标准
1. 可演示成交闭环 + 学习闭环 + 兜底闭环。
2. Gate 未通过时无法发布正式版本。
3. T+7 前不可发放分润。
4. ETA 超时自动升级官方托底。
5. 全链路有可查询审计记录。

---

## 10. 里程碑建议
- M1：成交主链路 + 基础结算
- M2：Learning Gate + Skill 升级
- M3：兜底三件套 + 担保退坡
- M4：监控告警 + 运营SOP + 上线复盘

---

## 11. 风险与缓解
- 风险：规则被绕过 -> 缓解：统一校验器 + 审计
- 风险：账务争议 -> 缓解：状态机 + 对账报表
- 风险：L1 流失 -> 缓解：兜底流程强制化
